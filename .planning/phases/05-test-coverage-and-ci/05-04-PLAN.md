<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| fork PR → CI runner | Untrusted PR code executes in CI. Mitigated by using `pull_request` (read-only token, no secrets) rather than `pull_request_target`. |
| CI runner → external APIs | None — live tests are deselected via `-m 'not live'`; CI needs no API keys. |
| third-party GitHub Actions → CI runner | checkout / setup-python actions run in the CI environment. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Elevation of privilege | CI trigger event | mitigate | Use `pull_request` (NOT `pull_request_target`): fork PRs run with a read-only token and no access to repo secrets, so malicious PR code cannot exfiltrate secrets or push. |
| T-05-08 | Information disclosure | API keys in CI logs | mitigate | The test step runs `pytest -m 'not live'`; no API keys are configured or required in CI, so they cannot leak into Actions logs. |
| T-05-09 | Tampering | third-party GitHub Actions | mitigate | actions/checkout@v4 and actions/setup-python@v5 are pinned to known major versions per D4 (no `@master`/floating refs). |
| T-05-SC | Tampering | npm/pip/cargo installs | accept | CI installs only the already-declared `[dev]` extra (pytest, pytest-asyncio, ruff) plus runtime deps already in pyproject.toml; no NEW package introduced. No legitimacy checkpoint required. |
</threat_model>
